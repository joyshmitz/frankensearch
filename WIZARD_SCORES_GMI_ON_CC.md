# WIZARD SCORES: GMI (Gemini 3.1 Pro) on CC (Claude Fable)

## Idea 1: Cross-Process Writer Lock + "Readers Never GC"
- **Score:** 980
- **Justification:** Claude Fable correctly identified a catastrophic data-loss scenario in the design of record. The plan (§11) dictates an in-process `asupersync::sync::Mutex` for writer coordination and a strict crash-only open (§11.4) that garbage-collects unreferenced segment files. In a multi-agent environment where an `fsfs search` (reader) runs concurrently with `fsfs index --watch` (writer), the reader will delete the writer's sealed-but-unpublished segments, permanently breaking the index. The proposed split between reader-open (no GC) and writer-open (advisory lock + GC) fixes this elegantly without new dependencies.
- **Strongest point in its favor:** It identifies and patches a silent multi-process data destruction bug that the current crash matrix (quill-e3.9) would never catch because the matrix is single-process.
- **Sharpest flaw:** Advisory lockfiles with staleness heuristics are notoriously race-prone on suspended processes (e.g., SIGSTOP'd agent workflows). A heartbeat delay could trigger a false takeover, leading to two writers thinking they hold the lock.
- **What would change my score:** If the plan had actually dictated OS-level `flock()` support, this wouldn't be as necessary, but given the strict dependency constraints, this is a near-perfect mitigation.

## Idea 2: "Skeleton-First" resequencing
- **Score:** 850
- **Justification:** Resequencing the bead graph to deliver a scalar, delta-free reference engine (G1a) before integrating SIMD and complex pruning kernels is a masterstroke in risk management. By decoupling the query parser (e4.1) and gauntlet runner (e6.2) from the SIMD tokenizer (e1.1), the differential testing infrastructure can start yielding conformance evidence months earlier. It forces the optimized kernels to land as "one-lever-per-change" perf beads against a green baseline, honoring the project's ledger discipline.
- **Strongest point in its favor:** It unblocks the differential gauntlet (E6) immediately, ensuring the highest-risk item (rank-parity and parser corner cases) is tested against actual execution rather than waiting for E1-E4 optimization beads to finish.
- **Sharpest flaw:** A scalar-only skeleton might bake in interface assumptions that require significant refactoring when integrating the columnar-radix flush (e1.7) or MaxScore/BMW heaps (e4.4), potentially thrashing the `Cursor` traits later in the project.
- **What would change my score:** If the `Cursor` and `Accumulator` trait boundaries in the skeleton aren't designed with the final vectorized/pruned architectures in mind, the rewrite cost would lower this score significantly.

## Idea 3: Shadow-Oracle mode in fsfs
- **Score:** 920
- **Justification:** Implementing a shadow dual-run mode for `LexicalSearch` provides the ultimate conformance test by subjecting Quill to the chaotic, adversarial queries generated by actual agents in the fleet. Fable smartly preserves the fusion `poll_immediate` contract (§9.1) by running the shadow engine in a bounded, asynchronous background task. This generates high-value, real-world divergence artifacts at zero risk to the user's immediate search experience.
- **Strongest point in its favor:** It socially and technically de-risks the default-flip commit (e7.6) by providing a continuously updating ledger of real-world conformance, far exceeding the coverage of synthetic E6 nightly generators.
- **Sharpest flaw:** The shadow mode effectively doubles the I/O and memory bandwidth pressure on the system, which could degrade the performance of the serving engine (tantivy) in resource-constrained agent environments, leading to spurious timeout errors.
- **What would change my score:** A higher score if it included a mechanism to dynamically shed shadow load when system metrics (CPU/IOps) cross a critical threshold, preventing any interference with the serving tier.

## Idea 4: Auto-minimizing divergence shrinker
- **Score:** 890
- **Justification:** The operational bottleneck for the E6 gauntlet will absolutely be diagnosing the 1M-doc nightly divergences. Building a delta-debugging (ddmin) loop to automatically shrink failures into 3-doc reproducible fixtures transforms a tedious, multi-hour agent debugging task into a matter of minutes. Tying this to an explanation-driven auto-triage that diffs BM25 factor vectors directly addresses the divergence classification requirements in §15.6.
- **Strongest point in its favor:** It leverages the deterministic guarantees of the engine (§1.5) to automate the most human-intensive part of search engine validation: root-cause analysis of floating-point score flips.
- **Sharpest flaw:** Automatic query shrinking might over-minimize the query to the point where the original bug's semantic context is lost, making it harder to understand *why* a complex parser edge case was triggered.
- **What would change my score:** If the minimizer explicitly preserved the original query AST in the `.quill-shadow/divergences.jsonl` artifact alongside the minimal query, it would be flawless.

## Idea 5: Fuel-metered, cancel-aware query execution
- **Score:** 750
- **Justification:** Fable correctly observes that wrapping a fully synchronous read path in a ready future (§9.1) violates the spirit of the `Cx` cancellation constitution (§1.3), leaving rayon threads vulnerable to pathological queries. Interleaving cooperative cancellation checkpoints and deterministic query fuel bounds this worst-case behavior. Returning `Err(Cancelled)` legally satisfies the `poll_immediate` contract without requiring the future to yield `Pending`.
- **Strongest point in its favor:** It prevents adversarial queries (like massive globs or dense phrase searches) from hanging the entire searcher region and provides a bit-reproducible fuel exhaustion error suitable for LabRuntime.
- **Sharpest flaw:** Implementing granular cooperative checkpoints inside the tightest inner loops (e.g., MaxScore decoding or SWAR snippet windowing) introduces branch overhead that directly threatens the sub-15ms p99 latency target (QG-6).
- **What would change my score:** The score would increase if the checkpoints were strictly limited to coarse-grained boundaries (e.g., per-segment merge or dictionary prefix-block edges) to guarantee zero impact on the `wide` unpack hot loops.
